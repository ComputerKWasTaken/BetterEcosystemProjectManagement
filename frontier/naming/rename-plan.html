Migration Plan

Frontier → Ultrascripts

Comprehensive rename plan across the BetterDungeon extension, project management docs, public-facing guides, and test suites.

01 — Scope

What gets renamed

The rename touches every reference to “Frontier” as BetterDungeon’s two-way communication system. This includes documentation, source code, story card titles, UI strings, test suites, and public-facing guides. It does not touch AI Dungeon’s own “Frontier” update — that is their name and stays as-is.

02 — Card titles

Story card title migration

The most critical rename surface. Story card titles are the protocol — they are how scripts and the extension communicate. Every script that uses Frontier cards will need to update these references.

Before
frontier:heartbeat
frontier:state:<name>
frontier:out
frontier:in:<module>
frontier:test:<module>
→
After
ultrascripts:heartbeat
ultrascripts:state:<name>
ultrascripts:out
ultrascripts:in:<module>
ultrascripts:test:<module>

Breaking change. Existing scripts that reference frontier:* card titles will stop working after the rename unless they are updated. Consider shipping a compatibility period where both prefixes are recognized, or provide a migration script/guide for scenario authors.

03 — Phases

Phased execution plan

The rename should be executed in phases to minimize risk and allow for testing between each step. Each phase is independently deployable.

Phase 1 BetterDungeon extension source code High risk

The core rename. This is where the runtime lives and where card titles are generated.

  • Rename services/frontier/ directory → services/ultrascripts/
  • Rename all source files: core.js, module-registry.js, ops-dispatcher.js, ws-stream.js, ws-interceptor.js, envelope.js
  • Rename features/frontier_feature.js → features/ultrascripts_feature.js
  • Update all frontier: string literals to ultrascripts: in card title generation
  • Update window.Frontier global → window.Ultrascripts
  • Rename FrontierModule interface → UltrascriptsModule
  • Rename FrontierContext interface → UltrascriptsContext
  • Update heartbeat payload: .frontier key → .ultrascripts
  • Update frontierCall() helper → ultrascriptsCall()
  • Update frontierHeartbeat() → ultrascriptsHeartbeat()
  • Update all import paths referencing services/frontier/
  • Update manifest.json if it references frontier files
  • Update popup UI: “Frontier” tab/section → “Ultrascripts”
  • Update popup CSS classes if namespaced to frontier

Test gate: After this phase, run the full extension locally and verify heartbeat writes ultrascripts:heartbeat, modules register correctly, and request/response cycle works end-to-end.

Phase 2 Test suites Medium risk

Update all AI Dungeon test scripts to reference the new card titles and variable names.

  • Update Provider AI test suite: all frontier:* card references → ultrascripts:*

  • Update SDK test suite card references

06 — Risks

Risks and mitigations

RiskImpactMitigation
Existing scripts break Any published scenario using frontier:* cards stops working Ship a compatibility period where both prefixes are recognized. Document migration clearly.
Missed references Runtime errors from leftover frontier strings in code Run exhaustive grep across all repos after each phase. Automated CI check for the old prefix.
Heartbeat payload key change Scripts reading .frontier from heartbeat will get undefined Include both keys (.frontier and .ultrascripts) during transition, then deprecate the old key.
Test suite drift Renamed test scripts no longer match live AI Dungeon scenarios Run full regression suite on live AI Dungeon after each rename phase.
Mobile sync Mobile and desktop extensions use different prefixes during rollout Coordinate mobile rename in Phase 5 before any public announcement.
07 — Summary

Execution summary

The rename from Frontier to Ultrascripts is a 6-phase migration touching two primary repositories, one mobile wrapper, one public website, and an unknown number of community-authored scenario scripts.

Phase 1 (extension source) is the critical path. Phases 2–5 are lower risk and can proceed in parallel. Phase 6 (community migration) should begin as soon as Phase 1 ships to a release.

The recommended approach is to ship the rename in a single coordinated release rather than rolling it out incrementally. This avoids a period where the extension uses one name and the docs use another. Plan for a compatibility layer in the card title parsing to gracefully handle scripts that have not yet migrated.
