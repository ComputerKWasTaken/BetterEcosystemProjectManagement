# AI Dungeon â€” CSS Architecture

> Reference for the atomic CSS class naming system, theme tokens, font families, and design patterns used across the AI Dungeon front-end.

## Overview

AI Dungeon uses **Tamagui** (a React Native Web UI framework) which generates **atomic CSS classes** â€” each class maps to exactly one CSS property-value pair. This is fundamentally different from traditional CSS where a class might represent a component (e.g., `.btn-primary`). Instead, every element has a long list of single-purpose classes.

This has major implications for BetterDungeon:
- **Class names are auto-generated and volatile** â€” they may change between deployments
- **Prefer `aria-label`, `role`, and `id` for selectors** over class names
- **Inline styles are common** â€” many properties are set via `style=""` rather than classes
- **Theme variables drive appearance** â€” colors, spacing, and sizing use CSS custom properties

## Semantic / Component Classes

These classes identify the type of component rather than a specific style:

| Class | Type | Description |
|-------|------|-------------|
| `is_View` | Layout | A generic container (React Native `View`) |
| `is_Row` | Layout | A horizontal flex container |
| `is_Column` | Layout | A vertical flex container |
| `is_Button` | Interactive | A button component |
| `is_ButtonText` | Text | Text inside a button |
| `is_Text` | Text | A text element (React Native `Text`) |
| `is_Input` | Input | An input/textarea component |
| `is_TextArea` | Input | Specifically a textarea |
| `is_ScrollView` | Scroll | A scrollable container |
| `is_Theme` | Theme | A theme boundary element |
| `is_LinearGradient` | Visual | A gradient background element |
| `is_ViewportWrapper` | Layout | Viewport-level wrapper for toasts/overlays |
| `is_ToastViewport` | Layout | Toast notification container |

**BetterDungeon note**: These `is_` classes are the most useful for identifying component types in selectors, combined with `aria-label` for specificity.

## Theme System

### Theme Class Hierarchy

```html
<span class="t_dark">                          <!-- Root theme: dark or light -->
  <span class="t_dark is_Theme">               <!-- Theme boundary -->
    <span class="font_body">                   <!-- Font context -->
      <span class="t_sub_theme t_core1 is_Theme">  <!-- Sub-theme -->
        <!-- Content uses core1 color tokens -->
      </span>
    </span>
  </span>
</span>
```

### Known Theme Classes

| Class | Type | Usage |
|-------|------|-------|
| `t_dark` | Root theme | Dark mode |
| `t_light` | Root theme | Light mode |
| `t_core1` | Sub-theme | Primary content area |
| `t_coreA1` | Sub-theme | Secondary/muted elements (undo, redo, settings buttons) |
| `t_gameplayPrimary` | Sub-theme | Primary gameplay accent (Take a Turn button) |
| `t_gameplaySecondary` | Sub-theme | Secondary gameplay (Continue, Retry, Erase buttons) |
| `t_primary` | Sub-theme | Primary accent color (Submit button) |
| `t_sub_theme` | Marker | Indicates a sub-theme wrapper |
| `is_Theme` | Marker | Theme boundary element |

### Theme Variables

Theme classes set CSS custom properties that child elements reference:

```css
/* These are set by the theme class and consumed by child elements */
var(--color)          /* Text color */
var(--background)     /* Background color */
var(--borderColor)    /* Border color */
```

### Theme Initialization

Theme is set on page load via an inline script:

```javascript
// Reads from localStorage, defaults to 'dark'
var e = localStorage.getItem('theme');
// Sets document.documentElement.classList to 't_dark' or 't_light'
```

## Font Families

AI Dungeon defines several font contexts via classes:

| Class | Usage | Description |

| `font_body` | Default | Standard UI text â€” inherits system font stack |

## CSS Specificity Notes for BetterDungeon

1. **Atomic classes use `:root` prefix**: e.g., `:root ._w-10037 { width: 100% }`. This gives them specificity of `(0, 1, 1)`.

2. **`!important` is used sparingly** but present on some utility classes (e.g., `pointer-events`):
   ```css
   .r-633pao { pointer-events: none !important; }
   .r-105ug2t { pointer-events: auto !important; }
   ```

3. **To override styles from BetterDungeon**, use either:
   - Inline styles (`element.style.property = value`) â€” highest specificity
   - Class with `!important` â€” matches or beats existing specificity
   - Targeting via `id` selector â€” `#game-text-input { ... }` has higher specificity than class-based rules

4. **Theme variables cascade** through the `is_Theme` boundary elements. Changing a variable on a parent `is_Theme` wrapper will affect all children.
